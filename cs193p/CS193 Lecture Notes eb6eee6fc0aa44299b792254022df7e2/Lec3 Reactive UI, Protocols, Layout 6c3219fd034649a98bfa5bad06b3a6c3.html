  • Protocol

    a protocol is sort of a "stripped-down" struct / class

    • It has functions and vars, but NO implementation (or storage)
    protocol Moveable {
	func move(by: Int)
	var hasMoved: Bool { get }
	var distanceFromStart: Int { get set }
}

// ... now any other types can claim to implement Moveable

struct PortableThing: Moveable { // comforms to Moveable protocol
	// MUST implement move(by:), hasMoved & distanceFromStart here
}

    Protocol Inheritance

    protocol Vehicle: Moveable {
	var passengerCount: Int { get set }
}

// ... implementing multiple protocols
class Car: Vehicle, Impoundable, Leasable {
	// MUST implement everything from Vehicle, Impoundable, Leasable here
}

    Protocol Extension

    struct Tesla: Vehicle {

}

extension Vehicle {
	func registerWithDMV() { /* implementation here */ }
}

    Why protocols?

    • A way for types (struct/class/other protocols) to say what they are capable of
    • For other code to demand certain behaviour out of another type
    • "functional programming": formalizing how data structures in our application function
    • Focus on the functionality and hide the implementation details behind it
    • The promise of encapsulation from OOP (taken to a higher level)

  • View Layout

    üí° How is the space on-screen apportioned to the Views?

    1. Container Views "offer" space to the Views inside them
    1. Views then choose what size they want to be
    1. Container Views then position the Views inside of them

    üí° Container Views:

    • The "stacks" (HStack, VStack, ZStack) devide up the space offered to them amongst their subviews
    • ForEach defers to its container to lay out the Views inside of it
    • Modifiers (e.g. .padding()) essentially "contain" the View they modify. Some do layout

    HStack & VStack

    Stacks divide up the space that is offered to them and then offer that to the Views inside.

    It offers space to its "least flexible"(with respect to sizing) subviews first.

    Image : inflexible (fixed size)

    Text : slightly more flexible

    RoundedRectangle : very flexible

    After the Views inside the stack choose their own size, the stack sizes itself to fit them.

    Spacer(minLength: CGFloat)

    Always takes all the space offered to it, but draws nothing

    Divider()

    Draws a dividing line cross-wise to the way the stack is laying out

    .layoutPriority(Double)

    • Stack's choice of who to offer space to next can be overridden with
    • It trumps "least flexible"
    HStack {
	Text("Important").layoutPriority(100)
	Image(systemName: "arraw.up") // the default layout priority = 0
	Text("Unimportant")
}

    If a Text doesn't get enough space, it will elide (i.e. "Import...")

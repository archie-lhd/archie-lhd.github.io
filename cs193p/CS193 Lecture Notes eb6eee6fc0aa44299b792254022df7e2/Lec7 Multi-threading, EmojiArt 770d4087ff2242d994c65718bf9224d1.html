  • Color vs. UIColor

    Color

    • Is a color-specifier, e.g., .foregroundColor(Color.green)
    • Can also act like a ShapeStyle, e.g., .fill(Color.blue)
    • Can also act like a View, e.g., Color.white can appear wherever a View can appear.
    • Due to this multifaceted role, its API is limited mostly to creation/comparison.

    UIColor

    • Is used to manipulate colors.
    • Also has many more built-in colors than Color, including ‚Äúsystem-related‚Äù colors.
    • Can be interrogated and can convert between color spaces.
    • (For example, you can get the RGBA values from a UIColor.)

  • Image vs. UIImage

    Image

    • Primarily serves as a View
    • Is NOT a type for vars that hold an image (i.e. a jpeg or gif or some such). That‚Äôs UIImage.
    • Access images in your Assets.xcassets by name using Image(_ name: String).
    • Many system images available via Image(systemName:). (SF Symbols)
    • You can control the size of system images with .imageScale() View modifier. System images are also very useful as masks (for gradients, for example).

    UIImage

    • Is the type for actually creating or manipulating images and storing in vars.
    • Very powerful representation of an image.
    • Multiple file formats, transformation primitives, animated images, etc.

  • Explanation of Multi-threading

    Don't Block my UI! Never let your UI to be unresponsive to the user.

    • Elaboration on Threads ...

      To keep our UI responsive when these long-lived tasks are going on, we execute them on a different ‚Äúthread of execution‚Äù than the UI is executing on.

      Most modern operating systems (including iOS) let you specify a ‚Äúthread of execution‚Äù to use. These threads appear to be all executing their code simultaneously.

      The challenge is to make multithreaded code authorable, readable and understandable.

    Queues

    Swift manages this complexity this using queues.

    In Swift, you are concerned only with queues. The system takes care of providing / allocating threads to execute code off these queues.

    Main Queue

    • The most important queue in iOS
    • Any time we want to do something in the UI, we must use this queue.
    • The system uses a single thread to process all the blocks of code from the main queue.
    • So it can also be used to synchronize. (e.g. to protect data structures by only mutating them on the main queue)

    Background Queues

    • Running "in paraller" (aka. simultaneously) with the main UI queue
    • You can influence the priority of these background queues by specifying a "quality of service" for that queue
    • The priority always < Main Queue

    GCD

    The base API for doing all this queue stuff is called GCD (Grand Central Dispatch).

    It has a number of different functions in it, but there are two fundamental tasks ...

    1. getting access to a queue
    1. plopping a block of code on a queue

  • Implementation of Multi-threading

    Creating a Queue

    DispatchQueue.main // the queue where all UI code must be posted
DispatchQueue.global(qos: QoS) // a non-UI queue with a certain quality of service

    ... where qos (quality of service) is one of the following:

    .userInteractive  // do this fast, the UI depends on it!
.userInitiated    // the user just asked to do this, so do it now
.utility          // this needs to happen, but the user didn‚Äôt just ask for it
.background       // maintenance tasks (cleanups, etc.)

    Plopping a Closure onto a Queue

    let queue = DispatchQueue.main
// or 
let queue = DispatchQueue.global(qos:)

queue.async { /* code to execute on queue */ }
queue.sync { /* code to execute on queue */ }
    • .sync blocks waiting for that closure to be picked off by its queue and completed. Thus, you would NEVER call it in UI code. You might (but still rarely) call it on a background queue. (e.g. to wait for some UI to finish)
    • .async will execute that closure "sometime later". (Whenever that closure gets to the front of the queue you plopped it onto.)

    Nesting

    DispatchQueue(global: .userInitiated).async {

	/*
		Do something that might take a long time
		This will NOT block the UI because it is happening [off the main queue]
		
		Once this long-time thing is done, it might require a change to the UI
		But we can‚Äôt do UI here because this code is executing off the main queue
		So, we just plop a closure with the UI code we want [onto the main queue]
	*/		

	DispatchQueue.main.async {
		// UI code can go here! we‚Äôre on the main queue!
	}
}
